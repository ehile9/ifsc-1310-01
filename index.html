Thoughts on Progressive Enhancement

Progressive enhancement is a best practices approach to inclusive web development. It is the flip side to the graceful degradation approach of building websites/applications that work on the newest, most up to date browser or platform with the intention of creating patches or fixes for outdated browsers. Unfortunately, the latter approach usually leaves the older versions with poor or inoperable sites. This also seems to be more cumbersome for the developers than a progressive enhancement approach. Progressive enhancement begins with a basic platform that will work well across all browsers with additional enhancements layered in to change according to the level of sophistication on the end users interface, giving a win/win situation for all (inclusivity). From what I have read it seems that it is much like the web standards of separating html, css, & javascript from one-another. It also seems to work as somewhat of an extension to accessibility standards as well. So why wouldn‚Äôt it just be the way things are done? It would seem to come back to teaching old dogs new tricks and short sided mindsets of expecting clients evolve to developers rather than the other way around. In the end, as a developer we should try to serve as large a community as possible while staying innovative which speaks to web standards, accessibility and progressive enhancement. Dragon Project Revisions And Reflections

Market Research and Feedback

For the first version of the Dragon Project I asked friends, family, and classmates for feedback on all four web pages of the Space-x Dragon Project. Initially criticism had to be drawn out of colleagues. But once the flow started it was hard shutting it down. That was not, however, the case when asking family members for critical analysis. Some of the feedback I got was that the music that autoplayed was annoying and that it would be better to load in the off position and let users turn on as they chose. Another concern was that you could not tell when you changed pages because the header section of the page consumed the entire viewable screen. The annoying music did change, so that let them know something changed but it was not clear that a new page loaded. Another point that was noted was the font color for the headers looked like you should have 3d glasses on. This was because I used a blue text color with a red shadow-box which resulted in difficult to read headings. Another stylistic issue was on my history page. The main text on the page was colored red with a space nebula set as the background. This was also very difficult to read.

Solutions & Iterations

To solve the issue of page navigation and user experience, I changed the audio file setting to be off when page loads and user can toggle music on with player controls. I also changed the height of the header element so that it did not take up the entire viewable screen which allowed user to understand that the page had indeed changed. In an effort to improve the readability I changed the border shadow on the headers and the font color and border shadow on the history page text. I also added the details and summary tags to the FAQ page and styled the toggle to a spaceship. On the homepage I added a gallery of several space-x images and incorporated them into a slide show with our new javascript techniques we learned on the last day of class.
